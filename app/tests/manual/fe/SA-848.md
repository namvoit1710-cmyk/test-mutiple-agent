# Manual Test Cases: SA-848 [FE]

### TC-001: Load Homepage - Happy Case

- **Priority:** Critical
- **Type:** Happy Case
- **Preconditions:** User is logged out or not authenticated.
- **Test Data:** N/A

| # | Step | Expected Result |
|---|------|-----------------|
| 1 | Navigate to the application's base URL (e.g., `/`). | The homepage should load successfully. |
| 2 | Verify the presence of key homepage elements (e.g., logo, navigation bar, main content area, footer). | All specified homepage elements should be visible and correctly rendered. |

- **Postconditions:** User is on the homepage.

- **Preconditions:** User is logged out or not authenticated.
- **Test Data:** N/A

| # | Step | Expected Result |
|---|------|-----------------|

### TC-004: Successful Login - Happy Case

- **Priority:** Critical
- **Type:** Happy Case
- **Preconditions:** User is on the login page.
- **Test Data:** Valid username, Valid password

| # | Step | Expected Result |
|---|------|-----------------|
| 1 | Enter a valid username in the username field. | The username should be accepted. |
| 2 | Enter a valid password in the password field. | The password should be accepted (masked). |
| 3 | Click the 'Login' button. | The user should be successfully authenticated and redirected to the application's dashboard or home page. |

- **Postconditions:** User is logged in and on the dashboard/home page.

### TC-005: Login with Empty Fields - Negative Case

- **Priority:** High
- **Type:** Negative Case
- **Preconditions:** User is on the login page.
- **Test Data:** Empty username, Empty password

| # | Step | Expected Result |
|---|------|-----------------|
| 1 | Leave the username field empty. | An error message indicating that the username is required should appear. |
| 2 | Leave the password field empty. | An error message indicating that the password is required should appear. |
| 3 | Click the 'Login' button with both fields empty. | The login attempt should be prevented, and error messages for both fields should be displayed. |

- **Postconditions:** User remains on the login page with error messages.

### TC-006: Login with Invalid Credentials - Negative Case

- **Priority:** Critical
- **Type:** Negative Case
- **Preconditions:** User is on the login page.
- **Test Data:** Invalid username, Invalid password

| # | Step | Expected Result |
|---|------|-----------------|
| 1 | Enter an invalid username in the username field. | The username should be accepted. |
| 2 | Enter an invalid password in the password field. | The password should be accepted (masked). |

- **Postconditions:** User remains on the login page with an error message.

### TC-007: Login with Special Characters - Edge Case

- **Priority:** Medium
- **Type:** Edge Case
- **Preconditions:** User is on the login page.
- **Test Data:** Username with special characters, Password with special characters

| # | Step | Expected Result |
|---|------|-----------------|
| 1 | Enter a username containing special characters (e.g., `!@#$%^&*()_+`) in the username field. | The username should be accepted. |
| 2 | Enter a password containing special characters (e.g., `P@$$wOrd!`) in the password field. | The password should be accepted (masked). |
| 3 | Click the 'Login' button. | If the credentials are valid, the user should log in successfully. If invalid, an appropriate error message should be shown. |

- **Postconditions:** User is logged in or remains on the login page with an error message.

### TC-008: Navigation between Pages - Happy Case

- **Priority:** Critical
- **Type:** Happy Case
- **Preconditions:** User is logged in.
- **Test Data:** Links to different sections of the app (e.g., Home, Dashboard, Settings).

| # | Step | Expected Result |
|---|------|-----------------|
| 1 | Navigate to the homepage. | Homepage loads correctly. |
| 2 | Click on a navigation link (e.g., 'Dashboard'). | The application should navigate to the Dashboard page, and the Dashboard content should be displayed. The URL should update accordingly. |
| 3 | Click on another navigation link (e.g., 'Settings'). | The application should navigate to the Settings page, and its content should be displayed. The URL should update accordingly. |

- **Postconditions:** User is on the navigated page.

### TC-009: Browser Back/Forward Navigation - Edge Case

- **Priority:** High
- **Type:** Edge Case
- **Preconditions:** User is logged in and has navigated through multiple pages.
- **Test Data:** Sequence of visited pages (e.g., Home -> Dashboard -> Settings).

| # | Step | Expected Result |
|---|------|-----------------|
| 1 | Navigate to Page A, then Page B, then Page C. | The URL and content should reflect the current page (Page C). |
| 2 | Click the browser's back button. | The application should navigate back to Page B, displaying its content and updating the URL. |
| 3 | Click the browser's forward button. | The application should navigate forward to Page C, displaying its content and updating the URL. |

- **Postconditions:** User is on the expected page after using browser navigation.
